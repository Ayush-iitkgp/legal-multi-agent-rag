Retrieval + Agent Interaction Flow

User: "What is the notice period for terminating the NDA?" ① Router Agent → LLM returns: { query_type: "fact_lookup", doc_targets: ["nda"] } → Maps "nda" → "nda_acme_vendor" → Sets is_risk_query = false ② Clause Extractor → Vector search: top_k = 8 (not risk) → Filter: keep only document_type == "nda_acme_vendor" → Dedup by (source, section_index, section_title) → Result: 3 unique NDA clauses ③ Answer + Risk Agent → LLM receives 3 clauses as [1], [2], [3] → is_risk_query = false → NO risk flags section → Output: "The notice period is 30 days[1]." → Citations used: [1] nda_acme_vendor.txt, section: Term and Termination

Risk Query Difference

top_k = 16 (2× fan-out)
No document_type filter — all agreements searched
Answer includes "Risk flags:" section

Safety Fallback

If routing filters remove all results, the system falls back to full unfiltered retrieval (still deduped) to prevent empty-context hallucination.

Why Section-Based Chunking & Nomic Embeddings

Section-Based Chunking

Legal text is structured — clauses have headers, numbered sections, defined boundaries
Fixed-size windows split mid-clause — "Party A shall... [chunk boundary] ...indemnify Party B" loses meaning
1 chunk = 1 complete obligation — preserves legal semantics intact
Clean citations — each chunk maps to a real section title (e.g. "Term and Termination")
Small corpus — 4 docs, ~20-30 sections total, no risk of oversized chunks

Header Detection

"1. Scope of Services" → numbered "GOVERNING LAW" → ALL-CAPS "## Definitions" → Markdown

Why nomic-embed-text?

Open-source — no API key needed, runs locally via Ollama
High quality — 768-dim embeddings, strong on MTEB benchmarks
Long context — 8192 token window, handles full legal clauses
Small footprint — ~274 MB, fits alongside the LLM in Docker
Matryoshka support — can truncate dimensions for speed

Alternatives Considered

all-MiniLMLower quality on legal text

Retrieval Strategy

Normal Queries (fact_lookup, cross_agreement)

top_k = 8
Retrieve 8 chunks from ChromaDB
Filter by document_type ∈ routed targets
Dedup by (source, section_index, section_title)
Typical result: 2–5 unique relevant clauses

Keeps context focused → better answer quality + lower token cost.

Risk Queries (risk_summary)

top_k = 16
2× fan-out for wider cross-agreement recall
No document_type filter — all agreements included
Same dedup logic applies
Ensures conflicting clauses across docs are surfaced

Risk analysis needs holistic view → wider retrieval window.

Why These Numbers?

8 balances recall vs. prompt length for a small corpus (~25 total chunks across 4 docs)
After filtering + dedup, typically 2–5 chunks reach the LLM
16 for risk ensures liability/indemnity clauses from multiple agreements aren't missed
Dedup prevents redundant context even with wider retrieval

Agent Prompts — Unified Answer + Risk

Why One Agent, Not Two?

Shared context — answer and risk analysis use the same retrieved clauses. Two agents would duplicate the context window.
Single LLM call — fewer round-trips, lower latency, lower cost. Critical for a local 1B model.
Conditional via flag — is_risk_query toggles the "Risk flags:" section. Not a separate agent, just a prompt switch.
Simpler graph — 3 nodes total. No conditional edges, no branching, easy to debug.

Prompt Toggle Logic

if is_risk_query: → "Add a 'Risk flags:' section with severity-tagged bullet points" else: → "Do NOT add any Risk flags section. Only answer + citations."

Prompt Structure

System role: "Legal contracts analyst for Acme Corp"

Instructions:

Use ONLY context clauses to answer
Inline bracket references: "30 days[1]"
End with "Citations used:" block
Format: [n] file.txt, section: title

Context fed to LLM:

[1] nda_acme_vendor.txt, section: Term and Termination <full clause text> [2] nda_acme_vendor.txt, section: Confidentiality <full clause text>

Evaluation Approach

LLM-as-a-Judge

Use a strong LLM (e.g. GPT-4) to evaluate answer quality on three axes:

CriterionWhat It Measures
FaithfulnessIs the answer grounded in the cited clauses? No hallucinated facts.
RelevanceDoes the answer directly address the question asked?
Citation accuracyDo the [n] references point to the correct source + section?

Each criterion scored 1–5. Judge prompt includes the question, retrieved context, and generated answer.

Recall@K

Measures retrieval quality independent of the LLM:

Recall@8 — for normal queries: what fraction of ground-truth clauses appear in the top-8?
Recall@16 — for risk queries: recall with wider fan-out

How to compute:

Recall@K = |relevant ∩ retrieved_top_k| / |relevant| Gold labels: manually annotated (question → relevant sections)

Evaluates retrieval separately from generation — isolates chunking and embedding quality.

Production Enhancements

Observability

LangSmith — end-to-end tracing: latency, token counts, prompts, retrieved docs per run
Already wired in codebase via env vars

DB Interaction

Move from file-based ingestion to a managed document store
Support uploads, versioning, audit trails

Auth & Rate Limiting

API key / OAuth for multi-user access
Per-user rate limits to control LLM costs

Tiered Models

Small model for Router (fast, cheap classification)
Large model for Answer (high-quality legal reasoning)
e.g. llama3.2:1b routes → GPT-4 answers

LLM-as-Judge Gate

Before returning answer, run a judge LLM
Check: grounded? relevant? well-cited?
Reject & regenerate if score < threshold

Semantic Caching

Cache (embedding → answer) for repeated / similar queries
Cosine similarity threshold to match cache hits
Saves LLM calls, cuts latency to near-zero for repeats
